# Phase 5 — SRS Support Design

> Status: Approved · Date: 2026-06-24 · Owner: Vivaldo
> Source: `docs/feedbacks/001-assumptions.md` (item B9)
> Scope: Fifth and final phase improving the `product-design-suite` plugin.

## Context

The suite models three first-class documents — PRD, SDD, ADR — each with a template (YAML
front-matter), a builder skill, a command, and integration into `pm-product-workflow`,
`traceability.js`, `pm-doc-sync`, and `pm-import`. An IEEE-830 **SRS has no native home**:
`pm-import` records it only as a read-only reference link, and `concepts.md §4` notes many
agile teams skip a formal SRS.

B9 surfaced the gap: teams that *do* keep an SRS (regulated / bank contexts) have nowhere to
put it, so it gets orphaned. This spec makes the SRS a first-class, **opt-in** fourth document.

This is **Phase 5** of the five-phase roadmap derived from the feedback:

1. Diagrams (B1+B2+B3+B10) — shipped.
2. Traceability (B4+B8) — shipped.
3. Template metadata & ADR relationships (B5) — shipped.
4. Authoring flow: import + derive-then-confirm (B6+B7) — shipped.
5. **SRS support (B9)** — this spec.

## Decisions (confirmed)

- **First-class SRS:** a dedicated `srs-template.md`, a `pm-srs-builder` skill, a `/pm-srs`
  command, and full wiring — parallel to PRD/SDD/ADR.
- **SRS owns detailed requirements:** the SRS is the canonical home for functional (`FR-NNN`)
  and non-functional (`NFR-NNN`) requirements. The PRD stays product-level (problem, goals,
  personas, scope, success metrics) and references the SRS's IDs.
- **Opt-in / backward-compatible:** with **no SRS**, the PRD owns `FR`/`NFR` exactly as today —
  zero behavior change. With an SRS, `FR`/`NFR` are canonical in the SRS.
- **Family split:** `FR` + `NFR` move to the SRS; **`BR` (business rules) and `UAT`
  (user-acceptance) stay in the PRD** — they are product/business concerns, not system-spec
  content.

## Mode detection (keystone)

**SRS mode is detected by the existence of `.product/srs/srs.md`** — the same file-existence
mechanism `pm-product-workflow` already uses to detect stages. No stored flag, no config:

- `.product/srs/srs.md` **absent** → **PRD mode**: current behavior, untouched.
- `.product/srs/srs.md` **present** → **SRS mode**: `FR`/`NFR` canonical in the SRS; the PRD's
  requirement sections reference it.

Every component (builders, workflow, traceability, doc-sync) keys off this one signal.

Because the PRD is normally authored *before* an SRS exists, `pm-srs-builder` carries a
**confirmation-gated migration**: when it runs against a PRD that already enumerates `FR`/`NFR`,
it lifts those rows into the SRS (reusing IDs verbatim), rewrites the PRD's §7/§9 as references
to the SRS, and leaves `BR`/`UAT` untouched. This reuses the established "propose-then-apply,
no silent rewrite" pattern. Creating the SRS is the trigger — no upfront mode question.

## Goal & Non-Goals

**Goal:** Give the suite a first-class, opt-in SRS. An `srs-template.md` provides an IEEE-830
home for detailed `FR`/`NFR`; `pm-srs-builder` authors it (and migrates requirements out of an
existing PRD); the workflow offers it between PRD and SDD; traceability, doc-sync, and import
become SRS-aware. The PRD-only triad keeps working unchanged when no SRS exists. Lock the
conventions with tests.

**Non-goals:** No change to PRD-mode behavior (no-SRS projects are untouched). No new runtime
dependency or parsing script (mapping stays skill-driven, consistent with Phases 3–4). No
diagram or OpenUI changes. No auto-migration without user confirmation. `BR`/`UAT` do not move.

## Component 1 — `srs-template.md` (C1)

`shared/templates/srs-template.md`, mirroring the other templates' YAML front-matter
(`title`, `status`, `version`, `owner`, `date`) and IEEE-830 structure:

1. **Introduction** — Purpose, Scope, Definitions/Acronyms, References (incl. link back to the
   PRD), Overview.
2. **Overall Description** — Product Perspective, Product Functions, User Characteristics,
   Constraints, Assumptions and Dependencies.
3. **Specific Requirements**
   - **External Interface Requirements** (user / hardware / software / communication).
   - **Functional Requirements** — the canonical `FR-NNN` table (ID, Requirement, Priority,
     Source, Acceptance Reference), matching the columns the PRD `FR` table uses today so
     migration is a row lift.
   - **Non-Functional Requirements** — the canonical `NFR-NNN` table, matching the PRD NFR
     columns (Category, Requirement, Target/Threshold, Measurement Method).
   - **Design Constraints / Standards Compliance.**
4. **Traceability** — note that the requirement coverage index lives in the SDD (generated by
   `traceability.js`); link to the PRD (`BR`/`UAT`) and SDD (`AR`).
5. **Appendices** (optional).

The `FR`/`NFR` tables are column-compatible with the PRD's existing §7/§9 tables so the
migration is a verbatim row move (IDs preserved).

## Component 2 — `pm-srs-builder` skill + `/pm-srs` command (C2)

`skills/pm-srs-builder/SKILL.md` (`name: pm-srs-builder`) and `commands/pm-srs.md`.

Responsibilities:

- Author `.product/srs/srs.md` from `srs-template.md`. Creating this file is what puts the
  project into SRS mode.
- **Own `FR`/`NFR` ID assignment** (stable across updates), mirroring how `pm-prd-builder` owns
  these families in PRD mode. Reuse source IDs verbatim when ingesting.
- **Questioning:** follow `questioning-protocol.md` — greenfield gap-question cadence by
  default; **derive-then-confirm mode** when authoritative source is provided (mapped content
  from `pm-import`, or source supplied by the user), exactly like the other builders.
- **PRD migration (confirmation-gated):** if `.product/prd/prd.md` already enumerates `FR`/`NFR`,
  propose lifting those rows into the SRS verbatim and rewriting the PRD's §7 Functional
  Requirements and §9 Non-Functional Requirements as references to the SRS. Show exact
  before/after; apply only on approval; never touch `BR`/`UAT`.
- On finalize, populate front-matter (bump `version`, refresh `date` on update); record
  unresolved gaps in Open Questions; suggest `pm-doc-sync`.

The skill sits between PRD and SDD in the lifecycle.

## Component 3 — Builder + workflow wiring (C3)

- **`pm-prd-builder`:** gains a mode branch — in **SRS mode** (`.product/srs/srs.md` exists), the
  PRD's §7/§9 reference the SRS rather than enumerating `FR`/`NFR`; `BR`/`UAT` are authored as
  today. In **PRD mode** (no SRS), behavior is unchanged. The builder does not move requirements
  itself — that migration is `pm-srs-builder`'s job; the PRD builder only honors the active mode.
- **`pm-sdd-builder`:** when an SRS exists, map `AR-NNN` to the SRS's `FR-NNN` (the canonical
  source) for traceability; when no SRS exists, map to the PRD's `FR-NNN`, unchanged. The
  reference is by ID, so the SDD text differs only in which document it cites.
- **`pm-product-workflow`:** the sequence becomes **PRD → (optional) SRS → SDD → ADR**. The
  detect-stage step gains: after a PRD exists and before the SDD, offer `pm-srs-builder` for
  teams that maintain a formal SRS (optional — skipping it keeps PRD mode). If `.product/srs/`
  exists, treat the SRS as the requirements source for the SDD stage. `.product/srs/` is created
  on demand by `pm-srs-builder` (the workflow need not pre-create it).

## Component 4 — `traceability.js` SRS-aware (C4)

- **`loadProduct(dir)`** also reads `.product/srs/srs.md` (empty string if absent) and returns
  it as `srs`.
- **`buildMatrix({ prd, sdd, adrs, srs })`** sources the requirement set by mode:
  - SRS present → `FR`/`NFR` parsed from the **SRS**; `BR` from the PRD; `UAT` from the PRD.
  - SRS absent → `FR`/`BR`/`NFR` from the PRD (current behavior, unchanged).
  - `AR` still parsed from the SDD; UAT-verifies and AR-traces links are ID-based and unchanged.
- The coverage index is still injected into the SDD §16. All linking is by ID, so reused
  `FR-NNN` IDs resolve across PRD/SRS/SDD regardless of which document defines them.

The default-arg shape stays backward-compatible (`srs = ''`), so existing callers and the
PRD-mode path behave identically.

## Component 5 — `pm-doc-sync`, `pm-import`, `concepts.md` (C5)

- **`pm-doc-sync`:** include the SRS in the impact graph — a changed SRS `FR`/`NFR` propagates to
  SDD `AR`, to PRD references, and to PRD `UAT`; a changed PRD reference points back to the SRS.
  Confirmation-gated as today.
- **`pm-import`:** an SRS source now maps to `srs-template.md` (drop the "SRS has no native
  template / linked read-only reference" language). The gap report gains an SRS target table.
  Source remains read-only; IDs reused verbatim.
- **`concepts.md`:** document the SRS as a supported optional document and its lifecycle slot
  (PRD → optional SRS → SDD → ADR), replacing the current "many teams skip a separate SRS"
  framing with "skip it (PRD owns requirements) or adopt it (SRS owns `FR`/`NFR`)."

## Testing

New `tests/srs-conventions.test.js` (mirroring the Phase 3/4 convention tests) asserts:

- `shared/templates/srs-template.md` exists with valid front-matter (`title`, `status`,
  `version`, `owner`, `date`) and documents the `FR`/`NFR` requirement tables and IEEE-830
  sections.
- `skills/pm-srs-builder/SKILL.md` exists with `name: pm-srs-builder` and a `description`, and
  its body documents: authoring `.product/srs/srs.md`, `FR`/`NFR` ID ownership, the
  derive-then-confirm branch, and the confirmation-gated PRD→SRS requirements migration.
- `commands/pm-srs.md` exists.
- `pm-prd-builder` and `pm-sdd-builder` document the SRS-mode branch (PRD references the SRS;
  SDD `AR` derives from SRS `FR` when present).
- `pm-product-workflow` documents the PRD → optional SRS → SDD → ADR sequence and offering
  `pm-srs-builder`.
- `pm-doc-sync` and `pm-import` document SRS handling; `pm-import` no longer says the SRS has no
  template.

Extend `tests/traceability.test.js`: with an SRS present, `FR`/`NFR` are sourced from the SRS
and `BR`/`UAT` from the PRD; with no SRS, behavior is unchanged (regression guard).

The Phase 1–4 suites must continue to pass, including `validate-plugin.test.js` (the new skill
must satisfy `name == dir`). Run with `node --test tests/*.test.js`.

## Open Questions

None outstanding. Four points were settled during design: (a) SRS is a full first-class document
(template + builder + command + wiring); (b) the SRS owns detailed `FR`/`NFR` while the PRD stays
product-level; (c) SRS is opt-in, detected by `.product/srs/srs.md` existence, with PRD mode
unchanged; (d) `FR`+`NFR` move to the SRS, `BR`+`UAT` stay in the PRD.
